## 3. Stack Depth / Back Behavior 분석

### 3-1. 확인된 스택 구조

```
[Depth 0] SmartThings 메인 앱 - 라이프 탭 (F-S21)
  └─ [Depth 1] Food 플러그인 진입 - Food 홈 (F-S01)
       ├─ [Depth 1] 검색 탭 (F-S19)       ← 탭 전환, 스택 push 아님
       ├─ [Depth 1] 커뮤니티 탭 (F-S20)   ← 탭 전환, 스택 push 아님
       └─ [Depth 2] 레시피 상세 (F-S14/F-S15/F-S16/F-S24)
            ├─ [Depth 2+Modal] 식사 시간 선택 다이얼로그 (F-S12)  ← Modal
            └─ [Depth 3?] 단계별 요리하기 (미수집)
       └─ [Depth 2] 오늘의 건강 데이터 (F-S13/F-S22/F-S25)
            └─ [Depth 3] 섭취 기록 목록 (F-S17)
```

### 3-2. 탭 전환 구조

- Food 하단 탭: 홈 / 검색 / 커뮤니티 / MY / 바코드(스캔) 5개 탭
- 탭 전환은 Back stack을 쌓지 않는 구조로 판단됨 (Back 시 Food 홈으로 복귀하지 않고 메인 앱으로 나갈 가능성)

### 3-3. Modal 구분

| 화면 | 타입 | 근거 |
|------|------|------|
| F-S12 (식사 시간 선택) | Modal Dialog | 배경 화면(레시피 상세)이 dimmed 상태로 유지되고 취소/확인 버튼 구조 |
| F-S17 (섭취 기록 목록) | Fullscreen 또는 BottomSheet | 하단 "완료" 버튼만 존재, 상단에 Back chevron 없음 → BottomSheet 가능성 높음 |

### 3-4. Back behavior 관찰

- F-S12 다이얼로그: "취소" → 레시피 상세 복귀, "확인" → 건강 데이터 화면으로 이동
- F-S17 섭취 기록: "완료" 버튼이 유일한 탈출 경로 (Back gesture 미확인)
- 레시피 상세 상단 "<" 버튼 존재 → 이전 화면(홈 또는 검색 결과)으로 복귀

## 4. SmartThings 메인 앱 ↔ Food 플러그인 경계 분석

### 4-1. 경계 진입점

| 구분 | 상세 |
|------|------|
| 진입 화면 | SmartThings 메인 앱 라이프 탭 (F-S21) |
| 진입 UI | 라이프 탭 내 "Food" 섹션 오른쪽 > 버튼 탭 |
| 진입 후 첫 화면 | Food 홈 (F-S01) - 상단 "<  Food" 헤더 노출 |

### 4-2. 구조적 차이점

| 항목 | SmartThings 메인 앱 (F-S21) | Food 플러그인 (F-S01, F-S18 등) |
|------|----------------------------|---------------------------------|
| 하단 탭 구성 | 홈 / 기기 / 라이프 / 자동화 / 메뉴 (5개) | 홈 / 검색 / 커뮤니티 / MY / 바코드 (5개) |
| 상단 헤더 | "Home ▼" 드롭다운 + + / ⋮ 아이콘 | "< Food" + 장바구니 / ⋮ 아이콘 |
| 상단 색상 | 흰색 배경 | 흰색 배경 (동일하나 탭바 구성이 완전히 다름) |
| Back 동작 | N/A (최상위) | < 버튼으로 메인 앱 라이프 탭 복귀 |

### 4-3. 플러그인 독립성 판단

- Food 플러그인은 **완전히 독립된 하단 탭 네비게이션**을 가짐
- 상단 헤더에 "< Food" Back 버튼 존재 → SmartThings 라이프 탭에서 push된 별도 Activity/Fragment로 판단
- 장바구니 아이콘(Food 전용)이 SmartThings 메인 앱 헤더에는 없음 → Food 플러그인 전용 기능
- WebView 여부: 스크린샷에서 Native UI 패턴(툴바, 탭바, 카드 레이아웃)이 관찰됨. 레시피 상세 내부는 WebView 혼합 가능성 배제 불가 (UI Hierarchy XML 없이는 확정 불가)

## 5. 미수집 화면 목록 (확정/보조/미확정 기준 적용)

> - **확정**: 실스크린샷 직접 판독 완료  
> - **보조 확인**: 외부 레퍼런스(스토어 메타데이터)로 기능 축만 확인  
> - **미확정**: 화면 라벨/CTA/전이가 직접 확인되지 않음

## 6. PRD 작성 시 활용 포인트

### 6-1. 혈당 기록 기능 진입 경로 설계 시

- **현재 칼로리 기록 진입 경로**: 레시피 상세 → "섭취한 음식으로 기록하기" → 식사 시간 선택 다이얼로그 → 건강 데이터 반영
- **혈당 기록 신규 진입 경로 후보**:
  - 옵션 A: 건강 데이터 화면 내 혈당 섹션 추가 → 별도 CTA ("혈당 기록하기")
  - 옵션 B: 식사 시간 선택 다이얼로그에 혈당 입력 필드 추가 (식후 혈당 동시 입력)
  - 옵션 C: MY 탭 내 건강 지표 허브 화면에서 혈당 기록 진입

### 6-2. 네비게이션 설계 시

- Food 플러그인은 독립적인 5탭 구조를 가짐. 혈당 관련 기능을 추가할 경우:
  - MY 탭 내 서브 섹션으로 수용 가능성 높음 (별도 탭 추가는 탭 과부하 위험)
  - 건강 데이터 화면(오늘의 건강 데이터)에 혈당 위젯 통합이 가장 자연스러운 위치

### 6-3. 기존 SmartThings 구조 영향도

- Food 플러그인은 SmartThings 라이프 탭에서 진입하는 독립 플러그인 구조
- SmartThings 메인 앱의 라이프 탭 Food 섹션(F-S21)에 혈당 관련 미리보기 위젯 추가 가능 (현재: "내게 맞는 요리" 레시피 썸네일 표시)
- 메인 앱 구조(홈/기기/라이프/자동화/메뉴) 변경 불필요

### 6-4. 플러그인 구조 확장 가능성

- 칼로리 기록 패턴(레시피 → 식사 시간 선택 → 건강 데이터 반영)이 이미 구현되어 있음
- 동일한 패턴으로 혈당 기록을 확장하려면:
  - "식사 시간 선택 다이얼로그"에 혈당 측정값 입력 필드 추가 또는
  - 별도 "혈당 기록" 다이얼로그/화면 신설
- 건강 데이터 화면의 날짜 타임라인(14~21일 슬라이더)은 혈당 추이 차트 배치에 적합한 기존 UI 패턴

### 6-5. 데이터 구조 관찰

- 섭취 기록의 단위: 음식명 + kcal + 인분(서빙) + 식사 시간(아침/점심/저녁/간식)
- 혈당 데이터 추가 시 필요한 필드: 측정 시간, 혈당 수치(mg/dL), 식전/식후 구분 → 현재 식사 시간 분류와 연동 가능
- 건강 데이터 화면에서 칼로리 목표(1616 kcal)가 설정값으로 존재 → 혈당 목표 범위 설정 UI도 동일 패턴으로 확장 가능

### 6-7. UX 패턴 일관성

- 주요 CTA 버튼은 모두 **오렌지색 Full-width 버튼** (식사 기록하기, 검색, 단계별 요리하기)
- 보조 액션은 **아웃라인 버튼 또는 텍스트 링크** (쇼핑리스트로 보내기, 노트 작성하기)
- 신규 혈당 기록 CTA도 동일 디자인 시스템을 따라야 함

핵심 목적은 “어디까지 확정이고, 어디부터 가설인지”를 명확히 구분하는 데 있다.

### 7-1. 근거 출처 (보조 근거)

|---|---|---|

- **보조 정보**: 스토어 메타데이터/매니페스트 기반 단서
